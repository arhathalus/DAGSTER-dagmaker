# Dagster backend × problem × DAG test matrix

`matrix.py` exercises every CDCL backend and helper mode against a range of
problem structures, each decomposed by several DAGs, and checks that they all
agree — then reports which backend/DAG is fastest per problem.

## What it checks

For each cell `(problem, DAG variant, mode)`:

1. **Verdict parity (hard gate).** The single-node DAG carries the full,
   undecomposed CNF, so its verdict is treated as ground truth. Every other
   cell must agree (SAT vs UNSAT). Disagreement = failure.
2. **Solution validity.** Every reported solution must satisfy the CNF
   (stdlib check, no PySAT).
3. **Completeness** (tiny instances only). Brute-forces all models and checks
   the enumerated solutions cover them.

Solution *counts* are **not** required to match across backends — pruning
generality differs, so counts are reported, not gated. (Same contract as
`tests/backend_equivalence.py`, which this complements.)

## Modes covered

| mode | backend | helpers |
|---|---|---|
| 0 | tinisat | — |
| 1 | tinisat | SLS (gnovelty) |
| 4 | minisat | — |
| 5 | cadical | — |
| 6 | cadical | SLS (gnovelty) |
| 7 | cryptominisat | — |
| 8 | minisat | SLS (gnovelty) |
| 9 | cryptominisat | SLS (gnovelty, prefix-feed only) |

SLS modes (1, 6) launch `master + worker + k gnovelty helpers`; non-SLS modes
launch `master + up to one worker per DAG node`, capped by the profile.

## Problems & DAGs

Problems are sourced from `Benchmarks/corpus/generators.py` (8 structured
classes), tiny hand-built SAT/UNSAT fixtures, and real instances (`su.cnf`).
Each problem gets DAG variants generated by `dagmake.py` at several `--nodes`
targets plus a `--search` (minimal-reporting) variant, always alongside a
guaranteed single-node baseline.

## Profiles

| profile | ranks | instances | timeout | runs where |
|---|---|---|---|---|
| `quick` | ≤6 | tiny + small | 30 s | locally (smoke) |
| `local` | ≤8 | tiny + small + medium | 60 s | locally (your 7–8-core box) |
| `hpc` | ≤128 | small + medium + large | 2 h | **emitted** as a SLURM array |

## Usage

```bash
# local regression (serial, your workstation)
python3 matrix.py --profile local          # -> results.csv + timing summary, exit 0/1
python3 matrix.py --quick                  # fastest smoke subset

# HPC: generate a SLURM job array (one array task per cell) — does NOT run here
python3 matrix.py --profile hpc --emit-hpc ./hpc
sbatch ./hpc/matrix_array.slurm            # on the cluster
```

The HPC emitter writes `matrix_array.slurm` (edit `--partition`/`--account` and
`srun` launcher to suit your scheduler), a `cells.tsv` manifest mapping array
task IDs to `(mode, ranks, k, dag, cnf)`, and materialised problems/DAGs under
`problems/`. The array launches each cell with `srun -n RANKS`, so it scales out
across the allocation; raise `sls_k`/`node_targets`/`max_ranks` in the `hpc`
profile in `matrix.py` for bigger scaling studies. CryptoMiniSat's internal
`set_num_threads` is a second (intra-node) parallelism axis worth sweeping there
as a future addition.

## Environment

Needs the built `../../dagster` binary, `mpirun`/`srun`, `LD_LIBRARY_PATH`
including `/usr/local/lib` (libcudd), and the repo `.venv` python for
`dagmake.py` (falls back to single-node DAGs if absent). Stdlib-only otherwise.
